# Threshold HBS Code Review Report

## 1. Scope

This report reviews the prototype implementation in `threshold_hbs.py`,
`demo.py`, `benchmarks.py`, and `README.md`, then compares the code against
the repository's own design claims and common coursework/research-code
standards.

## 2. Overall Assessment

The repository is a working prototype rather than a production-grade
cryptographic implementation. The main functional claim is supported:

- `demo.py` completes successfully for the minimal scheme and Extensions 1-5.
- `benchmarks.py` completes successfully and produces `benchmark_results.csv`.
- Verification succeeds in all provided demo and benchmark runs.

The implementation is therefore internally consistent with the README's
"prototype with progressive extensions" positioning.

## 3. Where The Code Matches The Claimed Design

### Minimal baseline

- The base scheme combines Lamport one-time signatures, Merkle authentication,
  and XOR-based share recombination.
- One-time key reuse is blocked through `used_leaves`.

### Extension 1

- The code realizes k-of-n signing through a k-of-k subset construction.
- Leaf ranges are assigned to fixed signer subsets, which is consistent with
  the README wording but narrower than a generic threshold scheme.

### Extension 2

- The distributed signing flow is implemented as an explicit two-round
  protocol.
- Helper-string lookup, session agreement, randomizer reconstruction, and
  CRV-based recovery are present.

### Extension 3

- Batch signing hashes messages into a batch Merkle root and signs the root.
- Each message is verified with an authentication path under the batch root.

### Extension 4

- Hierarchical batching is implemented with a subtree path plus an upper-tree
  path.
- Verification accepts either a standard Merkle path or a hierarchical path.

### Extension 5

- Winternitz chains replace Lamport secrets.
- The implementation preserves the two-round distributed threshold structure.
- Measured signature size and communication cost are substantially smaller than
  the Lamport-based distributed variants.

## 4. Gaps Against Paper-Style Rigor And Code Standards

### A. The exact paper is not identified

The current repository does not cite a paper, theorem numbering, section
mapping, or target parameter set. Because of that, the code can only be judged
against the repository's own claims, not against a specific external paper.

### B. Extension 2 is only partially distributed

The README says the fully trusted dealer assumption is "partially" removed, and
the code matches that wording. However, the implementation still depends on
central setup work:

- the dealer generates all leaves,
- the dealer assigns subsets,
- the dealer creates helper strings,
- the dealer precomputes CRV material.

So this is not a full distributed key generation design. It is a simulation of
distributed signing over centrally prepared material.

### C. Benchmark comparisons are not fully apples-to-apples

Different schemes are benchmarked under different parameter tuples. That makes
the trends useful, but it weakens strong cross-scheme claims such as "scheme X
is faster than scheme Y" unless the compared rows use the same
`(parties, threshold_k, tree_height, batch_size, w)` values.

### D. One dormant code-quality bug was fixed during this revision

`CRVEntry.to_dict()` previously referenced fields such as `self.parties`,
`self.tree_height`, and timing attributes that did not exist on `CRVEntry`.
That method has now been repaired so it returns entry-level metadata instead of
invalid attributes.

### E. Reproducibility documentation was too thin before revision

Before the README rewrite, the repository did not clearly state:

- that the project is a prototype,
- that Extension 1 is a subset-based realization,
- that Extension 2 remains centrally prepared,
- what files are generated by benchmarks,
- how to create visual analysis outputs,
- what limitations should be respected when interpreting results.

### F. Testing remains lightweight

The repo currently relies on demo runs and benchmarks as behavioral checks.
There is still no dedicated automated test suite for:

- invalid signer sets,
- leaf exhaustion edge cases,
- malformed authentication paths,
- tampered randomizers or shares,
- repeated signing attempts on the same leaf.

## 5. Empirical Findings From The Current Benchmarks

The benchmark run completed successfully on 2026-04-23 and produced these
high-level trends:

- Minimal and Extension 1 are the cheapest to set up and sign.
- Extension 2 adds clear overhead from two-round coordination and CRV logic.
- Extension 3 reduces average signing cost per message versus Extension 2.
- Extension 4 is structurally richer but not faster than flat batching in the
  best observed settings.
- Extension 5 dramatically reduces signature size and communication cost, but
  setup cost becomes the dominant expense.

The new `performance_analysis.py` script converts these benchmark results into
SVG figures and a compact Markdown summary.

## 6. Deliverables Added In This Revision

- `performance_analysis.py`
  Generates SVG charts and a Markdown summary from `benchmark_results.csv`.
- `README.md`
  Rewritten to better match the code, limitations, and reproducibility needs.
- `report.md`
  This review document.

## 7. Recommended Next Steps

If you want the repository to look closer to a paper-quality artifact, the next
three improvements should be:

1. Add a small automated test suite for tampering, edge cases, and one-time key
   reuse guarantees.
2. Add a "fair comparison" benchmark mode that runs multiple schemes under the
   same parameter tuples.
3. Expand lightweight API checks so dormant helper methods are validated too.
